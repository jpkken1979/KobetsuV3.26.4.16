## Fase 3: Seguridad

### Hallazgos confirmados

#### 1. Token admin en `localStorage` — **MEDIUM**

`src/lib/api.ts` lee `adminApiToken` desde `window.localStorage` y lo envía como header `x-admin-token` para rutas `/admin`.

**Riesgo:** en caso de XSS en frontend, el token puede exfiltrarse.  
**Contexto mitigante:** la app es interna/local-first y `server/middleware/security.ts` permite localhost sin token.

#### 2. Guard de admin bien endurecido para acceso remoto — **GOOD**

`server/middleware/security.ts`:

- exige `ADMIN_TOKEN` para acceso remoto a `/api/admin/*`
- usa `timingSafeEqual`
- limita rate de mutaciones
- añade headers de seguridad (`nosniff`, `DENY`, `Referrer-Policy`, `Permissions-Policy`)

#### 3. `.env` no está commitado — **GOOD**

No se detectó `.env` en git. Solo `.env.example`.

### Dependencias vulnerables

`npm audit --audit-level=high --json` devolvió **5 vulnerabilidades moderadas**:

1. **hono `< 4.12.14`** — advisory de HTML injection en `hono/jsx` SSR

### Evaluación práctica

### Hallazgos de mantenimiento

#### 1. El sistema está fuertemente acoplado al dominio de línea/fábrica — **EXPECTED BUT HIGH IMPACT**

La línea es la unidad operativa central. Cualquier cambio en configuración anual, responsables, rates o horarios impacta:

- selección de empleados
- cálculo contractual

- historial

Esto no es un bug, pero sí un **boundary crítico**: cambios sobre `factories`/config anual tienen alto radio de impacto.

Esto mejora recuperación, pero también mete costo operacional y side effects en flows calientes.
