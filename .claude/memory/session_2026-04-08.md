## Decisiones tecnicas

- **conflictDateOverride es nullable, no tiene un boolean separado en DB** — el flag `useConflictDateOverride` solo existe en el store Zustand (UI state). Si es null → usa factory.conflictDate. Simple y retrocompatible.
- **`??` en lugar de `||` para el resolver del PDF** — detectado en code review final. `||` trata `""` como falsy, inconsistente con la convención del proyecto.
- **Validación server-side usa effectiveConflictDate** — permite overrides más tardíos que la factory, que es el caso de uso principal (contratos históricos retroactivos).
- **Toggle movido a sección de fechas** — code review detectó que estaba mezclado con texto informativo del bloque verde de factory data.

## Descubrimientos

- `db:push` de Drizzle Kit falla en este proyecto cuando intenta recrear índices ya existentes (bug conocido de Drizzle). La migración se aplicó con `ALTER TABLE` directo. En una DB fresca funciona normal.
- `pdf-lib` no estaba en `package.json` aunque el código lo importaba dinámicamente — el error era solo TS, no runtime (pdf-lib estaba como dep transitiva).
- El `parseCompanySheet` en factory import propaga grupos de headers correctamente incluso sin merge cells visibles en ExcelJS — ExcelJS resuelve los merges automáticamente.
- `待機中` (7 empleados en el archivo real) se importaba como `active` — corregido a `onLeave`.
- `seed.ts` tiene un `CREATE TABLE contracts` manual que no se sincronizaba con el schema Drizzle — al agregar columnas nuevas hay que actualizarlo también.
