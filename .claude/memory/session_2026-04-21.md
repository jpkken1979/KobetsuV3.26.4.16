## Decisiones tecnicas

### `.env` versionado (excepcion documentada)
- **Porque**: repo privado, single-user, sin forks externos planeados. Costo de mantener `.env.example` + `.env` separado > beneficio marginal de seguridad en este contexto.
- **Guardarraiz**: si el repo se vuelve publico o hay fork externo, ejecutar `git rm --cached .env` + rotar tokens. Documentado en 3 lugares (CLAUDE.md, security.md, best-practices.md).

### Override esbuild en vez de esperar drizzle-kit
- **Porque**: el CVE afecta el dev server de esbuild (CORS laxo). Aunque el impacto practico en single-user es bajo, cerrar vulns en `npm audit` es barato via `overrides`.
- **Alternativa descartada**: esperar drizzle-kit upstream (PR abierto hace meses en su repo).
- **Riesgo evaluado**: `@esbuild-kit/esm-loader` solo usa esbuild para cargar archivos TS — probabilidad de ruptura < 5% con bump 0.24->0.25.
- **Verificado**: `drizzle-kit --version`, `drizzle-kit generate --help`, 762 tests pass, build OK.

### NO migrar `@hono/node-server@2` ahora
- **Porque**: publicado hace ~10 horas. Costo de un runtime roto (no poder generar contratos legales) >> beneficio de estar al dia.
- **Re-evaluar**: cuando v2 tenga >=3 semanas O salga v2.0.1 (patch inicial = bugs tempranos cazados).

## Descubrimientos

### drizzle-kit 0.31.10 (latest) sigue arrastrando @esbuild-kit
Se creia que la cadena vulnerable estaba solo en versiones viejas. No — incluso en la ultima drizzle-kit pull `@esbuild-kit/esm-loader@2.6.5` que a su vez pull `@esbuild-kit/core-utils@3.3.2` que pull `esbuild@<=0.24.2`.
El override es la unica forma de cerrar el CVE sin esperar upstream.

### npm `overrides` con caret conflicta con deps que piden rangos mayores
Primer intento fue `"esbuild": "^0.25.0"`, pero tsx pide esbuild >=0.26 y npm reporto "invalid" (tsx tiene copia propia en `node_modules/tsx/node_modules/esbuild`).
Fix: cambiar a rango abierto `">=0.25.0"` — permite cualquier version moderna y cierra el CVE.

### useReducedMotion — 100% cobertura (falso alarma del grep inicial)
Audit inicial dio 17/32 archivos con `useReducedMotion`. Delegado a subagente Explore: todos los archivos con animaciones propias (`whileHover`, `whileTap`) SI usan el hook. Los 15 aparentes gaps eran archivos que delegan a primitivas (`<Animated>`, `<DialogContent>`) que ya respetan reduce-motion.
