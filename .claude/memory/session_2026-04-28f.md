## Decisiones tecnicas

### Por qué mantener `companyName` reales en los .example
La lógica R11 en `server/services/takao-detection.ts` filtra por `companyName.includes("高雄")`. Los tests `takao-detection.test.ts`, `batch-helpers.test.ts`, `batch-contracts-service.test.ts` y `import-employees-service.test.ts` dependen de strings exactos como `高雄` y `瑞陵精機株式会社`. Renombrar esas empresas en los `.example` rompía 3+ archivos de tests y la feature de detección de re-entries en 1 año. Anonimizamos solo PII de personas físicas (`representative`, `supervisorName`, `commanderName`, `complaintClientName`, `complaintUnsName`, `managerUnsName`) y los datos de empleados completos. Los nombres de empresa son datos de registro mercantil japonés (公開済み — no PII).

### Por qué fallback en seed.ts y no copy script
Cualquier dev nuevo o CI hace `git clone + npm run db:seed:force` y obtiene seed válido sin intervención manual. Si en lugar de fallback hubiéramos requerido `cp *.example.json *.json` post-clone, sería un footgun. El warning explícito (`employees.json not found, using employees.example.json (synthetic data)`) deja claro qué seed está activo.

### Por qué PRNG seeded en anonymize-seeds.cjs
Mulberry32 con seed fija (20260428) garantiza que regenerar `.example.json` desde los mismos `.json` reales produce el mismo output byte-a-byte. Esto evita ruido en diffs si el script se vuelve a correr y permite que múltiples PCs del mismo dev generen idénticos `.example`.

### Por qué dejé la purga del histórico como acción manual
`git filter-repo` reescribe SHAs de todo el histórico. Cualquier clone existente queda inutilizable y debe rehacer `git clone`. Es un cambio coordinado que requiere notificación a otras PCs/colaboradores. El usuario tiene 2 PCs (laboral + personal) y posiblemente más clones, así que lo dejé documentado en ESTADO_PROYECTO.md como receta para ejecución manual cuando esté listo a coordinar.

## Descubrimientos

- **`.gitignore` siempre tuvo `.env` excluido** (líneas 14-17). Esto es un patrón conocido — la documentación oscila pero el archivo de configuración crítico mantuvo la postura segura por defecto.
- **Tests pasan con solo `.example.json`**: 781/781 (43 archivos) usando datos sintéticos. Esto valida que la suite no tiene dependencias ocultas en valores específicos de PII (employee numbers reales, birthDates específicas, etc.) — solo en los strings de companyName que decidimos preservar.
- **git detectó `companies.json` → `companies.example.json` como rename** (98% similar, porque solo cambia `representative` → null en 14 entries). Eso es bueno para legibilidad del log: un cambio que en bytes parece masivo aparece como rename en git.

### Acción manual del usuario — Purga del histórico
Los `.json` con PII todavía viven en commits anteriores a `ec5ea0d`. Para sacarlos del histórico:

```bash
git filter-repo --path data/seed/employees.json \
  --path data/seed/factories.json \
  --path data/seed/companies.json --invert-paths
git push origin --force --all
git push origin --force --tags
```

Requiere:
- Coordinación con cualquier otro clone del repo (otra PC, GH Actions, colaboradores)
- Backup completo antes (`git clone --mirror` a otra ubicación)
- Aceptar que SHAs cambian de raíz a HEAD

No se ejecutó automáticamente por el blast radius.
